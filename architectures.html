Reference Architectures

Operating models that hold up in real work

Choose architecture by business stage, risk profile, and team capacity. Good architecture reduces coordination waste and increases trust in agent output.

Architecture set for v1

Use these as starting points, then tune for your operating constraints.

A1

Solo Operator Setup

Best for individual operators and builders validating real utility quickly.

  • Planner + Worker + Reviewer in a single loop
  • Daily queue review and strict task boundaries
  • Manual approval for all external actions
A2

Founder + AI Chief of Staff

Best for founders handling strategy while delegating structured execution.

  • Founder-owned decision lane
  • Chief-of-staff agent for planning and coordination
  • Specialist workers by function (ops, growth, research)
A3

SMB Agent Support Stack

Best for SMBs needing operational consistency across customer and internal workflows.

  • Department-tagged work queues
  • Service-level definitions for response and completion
  • Reviewer role aligned with operations leadership
A4

CRM + Task + Calendar Stack

Best when customer context and execution timing drive business value.

  • CRM as context source
  • Task platform as execution ledger
  • Calendar as planning and follow-up anchor
A5

Browser Worker Stack

Best for process-heavy web operations that require reproducible execution traces.

  • Explicit browser worker guardrails
  • Step-based runbooks and snapshot evidence
  • Escalation on unknown UI states or policy conflict

Architecture selection matrix

Pick the lightest model that can still enforce quality and control.

Architecture Team size Best for Risk level
Solo Operator Setup 1 Rapid experimentation with constrained automation Low to moderate
Founder + AI Chief of Staff 1-5 Founder-led delegation with strong decision gates Moderate
SMB Agent Support Stack 5-50 Cross-functional operations and service consistency Moderate to high
CRM + Task + Calendar Stack Any Pipeline, follow-up, and execution reliability Moderate
Browser Worker Stack Any High-volume browser workflows with traceability needs High without controls

Implementation sequence

A practical rollout path that avoids over-automation too early.

Phase 1

Establish roles, templates, and review cycles with one contained workflow.

Phase 2

Add integrations and standard reporting. Keep a human gate on external actions.

Phase 3

Scale to multiple lanes with metric-driven tuning and monthly architecture reviews.
